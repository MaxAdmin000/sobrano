# Telegram-прокси через Cloudflare Worker

## Зачем

Хостинг сервера СОБРАНО (Selectel, Москва) блокирует прямой выход на `api.telegram.org` —
последствия блокировки Telegram в РФ. Из-за этого Telegram-бот молча не отправляет
уведомления о новых заказах админу.

Решение — поставить **прозрачный прокси** на Cloudflare Worker. CF не блокируется в РФ,
у Workers есть бесплатный план на 100 000 запросов/день (для магазина — с большим запасом).
Worker принимает запросы вида `/bot<TOKEN>/sendMessage`, форвардит на `api.telegram.org`,
возвращает ответ как есть. Шифрование HTTPS на всём пути, токен по сети не утекает.

## Шаги (5 минут)

1. Открой [dash.cloudflare.com](https://dash.cloudflare.com). Если аккаунта нет —
   регистрация бесплатна, карта не нужна.

2. В сайдбаре: **Workers & Pages → Create → Hello World → Get started**.

3. Имя Worker'а: например `sobrano-tg`. Жми **Deploy**.

4. После деплоя жми **Edit code**.

5. Удали весь дефолтный код, вставь содержимое файла
   [`docs/telegram-proxy.worker.js`](./telegram-proxy.worker.js) из этого репо.

6. Жми **Deploy** (правый верхний угол) → **Save and Deploy**.

7. Сверху появится URL вида `https://sobrano-tg.<твой-сабдомен>.workers.dev`.
   Скопируй его.

8. Открой админку → `https://sobrano.store/admin` → таб **Настройки** →
   карточка **Уведомления админу · Telegram** → поле **URL прокси** →
   вставь URL Worker'а (без слэша в конце). Жми **Сохранить**.

9. Тут же жми **Тестовое сообщение** — в чат должно прилететь
   «🌸 СОБРАНО · тест уведомлений».

Если не прилетело — проверь логи сервера:

```bash
ssh root@80.249.150.166 'journalctl -u sobrano-backend -n 30 --no-pager | grep telegram'
```

Сейчас в коде есть явное логирование `[telegram] …` для сетевых ошибок и не-ОК ответов
от API — по ним сразу видно в чём дело (неверный токен, неверный chat_id, бот не добавлен
в группу как админ и т.д.).

## Если что-то идёт не так

| Симптом | Что проверить |
|---|---|
| `[telegram] network error: ENETUNREACH` | сервер не достучался ни до прокси, ни до Telegram. Проверь URL прокси — должен быть `https://…workers.dev`, не `http://`. |
| `[telegram] api responded not-ok: chat not found` | chat_id неверный. Для группы — `-100<id>`. Узнай через [@userinfobot](https://t.me/userinfobot) или зайди в [@RawDataBot](https://t.me/RawDataBot). |
| `[telegram] api responded not-ok: bot was kicked` | бот удалён из чата или не имеет права писать. Добавь бота в чат и сделай админом (минимум — право «Отправлять сообщения»). |
| Worker отдаёт 502 / fetch_failed | у CF разовый сбой. Подождать минуту и повторить. Если стабильно — проверить, что URL Worker'а правильный. |

## Лимиты и стоимость

- Free-план CF: **100 000 запросов/день**, **10 ms CPU** на запрос (наш прокси — это ~1 ms).
- 1 заказ = 1–2 запроса к боту (newOrder + paid).
- Реальный расход при 500 заказов/день: ~1000 запросов. Free-лимит хватит на 100×.

Если когда-то упрётесь в лимит — Workers Paid даёт 10 000 000 запросов за $5/мес.

## Безопасность

- Бот-токен идёт по HTTPS внутри URL-пути (так же, как в прямой работе с Telegram API).
- CF Workers не логирует body/тело запросов по умолчанию.
- Worker фильтрует исходящие заголовки (`pickHeaders`): на api.telegram.org уходят только
  `content-type/content-length/accept/user-agent`, без `cf-*` / `x-real-ip` / `cookie`.
- Если хочется ещё параноику — добавь свой секрет в URL Worker'а
  (например, `/secret-prefix/bot<TOKEN>/…`) и обнови `proxyUrl` в админке так же.
